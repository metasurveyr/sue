Sobre el proyecto

  • Es mi proyecto de trabajo final de la Licenciatura en Estadística
  • Existe una versión previa llamada srvyuRu la cuál presenté en el LatinR 2022 y en el evento de GURU/RLadies de 2022, el mismo tenía muchas dependencias y era muy lento.
  • Superar algunas limitaciones de paquetes actuales para procesar encuestas especificas.
  • Combatir la limitaciones de paquetes actuales donde funciones que asisten a la creación de variables depende de la estructura de la encuesta y poder ser flexible para cualquier tipo de encuesta.
  • Empece a trabajar en la nueva versión 2023 se ha escrito desde cero y se ha mejorado la eficiencia y la simplicidad.

Sobre metasurvey

  • Diseñado para trabajar con el paquete survey y aquellos que tengan un diseño muestral como por ejemplo convey.
  • Se encuentra en una versión de desarrollo en GitHub
  • Actualmente el chequeo de errores se encuentra en 0 errores y 0 advertencias en R CMD check, en varias plataformas y versiones de R a partir de 4.0.0 y en R-devel en GitHub Actions.
  • La documentación se encuentra en metasurveyr.github.io/metasurvey en una versión mínima.
  • Las dependencias son mínimas entre ellas son data.table, survey, R6, httr, jsonlite, crayon, glue y bit64.

Que diferencia metasurvey del survey

  • Transparencia en la creación de variables
  • Facilidad para compartir y reproducir variables
  • El proceso se realiza en un solo lugar
  • Posibilidad de realizar estimaciones a nivel mensual, trimestral, anual o promedios móviles de ellos.
  • Diseño muestral flexible y fácil de definir
  • Se inspira en tidymodels para definir conjuntos de pasos steps y recipes conjunto de pasos con un sentido común.
  • Todo el flujo termina en un workflow que permite realizar estimaciones a nivel mensual, trimestral, anual o promedios móviles de ellos, para una sola encuesta o una serie de ellas.

Flujo básico con metasurvey

  • Carga de micro datos y diseño muestral. Pueden definirse diseños muestrales para diferentes periodos e incluir pesos replicados. Para esto puede utilizarse la funciones load_survey, load_panel_survey en conjunto con add_weight y add_replicate.
  • Creación de variables, en el cual se utilizaran los steps_compute y steps_recode. En cada paso se tiene información sobre cómo fue construida y las dependencias en el caso que se necesite.
  • El paso anterior se puede omitir si se cuenta con una receta previamente definida y compartida con la función get_recipe.
  • Hornear las variables. Se calculan las variables definidas en el paso anterior. Por defecto metasurvey tiene lazy_processing = TRUE. Esto puede modificarse con set_lazy_processing(FALSE) o hornear con bake_steps.
  • Poder compartir recetas con otros usuarios y reproducir los resultados. Con publish_recipe
  • Estimación de indicadores a nivel mensual, trimestral, anual o promedios móviles de ellos. Con workflow se le pueden indicar múltiples indicadores a estimar.

Utilidad de las recetas en metasurvey

  • Cada usuario con los steps genereados puede crear un conjunto de ellos, añadirle un nombre, descripción y un tópico y para que edición de la encuesta fue creada.
  • Las recetas pueden ser guardadas de forma local o publicadas con API de metasurvey desde el mismo paquete. Las recetas se almacenan en una base de datos de documentos llamada MongoDB.
  • Las recetas pueden ser descargadas por otros usuarios y reproducir los resultados.
  • La disponibilidad de las recetas en la API de metasurvey permite que sea la propia comunidad la que genere recetas y las comparta.
  • Instituciones pueden compartir sus propias metodologías de cálculo de variables y estimaciones, permitiendo facil acceso y un estandar de calidad.

API de metasurvey

API Mongo

¿Cómo está hecho metasurvey?.

  • El nombre hace referencia a “meta-programming” y “survey”.
  • Además de la meta-programación, se utiliza programación funcional y orientada a objetos, esta última para modelar los diferentes objetos que se utilizan en el paquete como lo son:
    • Survey para la encuesta, que contiene los datos y los metadatos. Estos son necesarios para obtener recetas desde la API de metasurvey (MongoDB).
    • PoolSurvey para procesar encuestas en paralelo con un sentido común, pueden estar indexados en el tiempo o no. Ejemplo: Ediciones de la misma encuesta
    • RotativePoolSurvey para procesar encuestas con paneles rotativos. Ejemplo: ECH luego de 2021.
    • Step para las variables, aquí es donde se encuentra la esencia de la meta-programación. Se guardan en este objeto como se construye un nueva variable, mostrando la forma de cálculo y se genera dependencias.
    • Recipe para las recetas, grupo de steps con un sentido común, el usuario puede compartir y descargar de internet.
    • Workflow para el flujo de trabajo y obtener los resultados, estimar ciertos indicadores a nivel mensual/trimestral/anual o promedios móviles de ellos. En esta clase se encuentra el diseño muestral cuya dependencia y sintaxis

Sobre la meta-programación

  • La meta-programación es una técnica de programación en la que un programa es capaz de manipular otros programas como si fueran datos.
  • Cada step es un objeto que contiene la llamada que se realizó para crear la variable en base a esta se puede desagregar variables que se crean en esa llamada y las dependencias que tiene.
  • Dentro de un workflow pueden utilizarse las mismas funciones que en survey sin indicarle el diseño muestral ya que se encuentra almacenado en el objeto donde se guarda la encuesta.

Red de metasurvey

Ejemplo: ECH con paneles rotativos

  • La Encuesta Continua de Hogares (ECH) es una encuesta que permite obtener información sobre el mercado de trabajo en Uruguay y otros indicadores sociales.
  • Con el objetivo de reducir margenes de error y una mayor apertura en ciertos grupos como aperturas geograficas y grupos demoográficos, se ha decidido implementar paneles rotativos.
  • La ECH puede verse como dos encuestas distintas:
    • Encuesta multipropósito con un diseño cross-section.
    • Encuesta de mercado de trabajo.

Esto esta extraído de los materiales relacionados en el sitio de la ECH. Portal ECH INE

Cocinar las variables

Aunque se han definido las variables, no se han calculado aún. Se puede usar la función bake_steps para calcularlas.

Ver el código 
ech_2023_bake <- bake_steps(ech_2023)
ech_2023$follow_up$ECH_01_23$steps[[1]]$bake
[1] FALSE
Ver el código 
ech_2023_bake$follow_up$ECH_01_23$steps[[1]]$bake
[1] TRUE

El concepto de lazy processing permite definir las variables y luego calcularlas en un paso posterior solamente cuando se necesiten. Si bien actualmente no existe un “catalizador” es rápido por el motor de data.table, pero en el futuro se podría implementar un catalizador.
